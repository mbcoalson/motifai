### 2. Designed Core Domain Models

#### CharacterProfile (Flexible, Story-Agnostic)
```python
class CharacterProfile(BaseModel):
    # REQUIRED: Just name
    name: str

    # OPTIONAL: Everything else
    role: Optional[str]
    basic_traits: Optional[List[str]]
    arc_stages: Optional[List[ArcStage]]
    voice_samples: Optional[List[VoiceSample]]
    contradictions: Optional[List[str]]
    sensory_filter: Optional[Dict[str, int]]
    regional_voice: Optional[str]

    # Extensible for story-specific needs
    class Config:
        extra = "allow"
```

**Key Design Decision**: Graceful degradation
- Minimal data = basic validation (better than nothing)
- Rich data = sophisticated validation (catches subtle voice drift)
- Writer decides complexity level

#### ArcStage (Voice Evolution Tracking)
```python
class ArcStage(BaseModel):
    stage_id: str  # "stage_1", "act_2_crisis", etc.
    name: str
    description: Optional[str]
    vocabulary_register: Optional[str]  # formal/informal/technical
    emotional_tone: Optional[List[str]]
    speech_patterns: Optional[List[str]]
    typical_phrases: Optional[List[str]]
    chapter_range: Optional[Dict[str, int]]  # start/end chapters
    date_range: Optional[Dict[str, date]]  # in-story dates
```

#### ValidationResult (Validation Output)
```python
class ValidationResult(BaseModel):
    character_name: str
    excerpt: str
    is_valid: bool
    confidence_score: float  # 0-1
    flagged_passages: Optional[List[Dict]]  # {text, reason, severity, suggestion}
    severity: Optional[str]  # critical/moderate/info/passed
    arc_stage_used: Optional[str]
    model_used: str
    summary: Optional[str]
    suggestions: Optional[List[str]]
```

### 3. Designed YAML Configuration System

**Story-agnostic structure**:
```
config/
└── stories/
    ├── water_rising/
    │   └── characters/
    │       ├── santiago_esposito.yaml  (full-featured)
    │       └── anais_non.yaml
    └── example_story/
        └── characters/
            └── example_character.yaml  (minimal)
```

**Example: Full-Featured (Water Rising)**
```yaml
name: Santiago Esposito
role: protagonist
basic_traits: [practical, blue-collar, skeptical]
regional_voice: Naples, FL - working-class vernacular
sensory_filter: {touch: 95, sound: 70, sight: 50}

arc_stages:
  - stage_id: stage_1
    name: Pre-transformation skeptic
    vocabulary_register: informal/working-class
    emotional_tone: [pragmatic, patient, reserved]
    speech_patterns: [economy of words, plain speech]
    typical_phrases: ["Water does what water does"]
    chapter_range: {start: 1, end: 5}

voice_samples:
  - text: "Tide's pulling harder than usual"
    context: Observing natural phenomena
    arc_stage: stage_1
```

**Example: Minimal (Any Writer)**
```yaml
name: Detective Murphy
role: protagonist
basic_traits: [gruff, cynical, drinks too much]
# That's it! System works with just this.
```

## Important Context for Next Session

### The "Why" Behind Design Choices

1. **Why optional fields?**
   - Some writers want minimal setup (just name)
   - Others want rich profiles (Water Rising level)
   - System should work for both, not force complexity

2. **Why separate Rogue and Consciousness?**
   - Different use cases: validation vs roleplay
   - Can share character data without coupling systems
   - Each can evolve independently

3. **Why defer Reflect to Phase 5?**
   - Need CLI tools working first
   - Auto-learning requires correction workflow in place
   - Phase 1-4 build foundation, Phase 5 adds intelligence
